# Provider Assumptions

This log records provider behavior that Linode Image Lab depends on. It is a
source-backed checklist for future behavior, docs, and test changes; it is not
a record of live API verification.

Checked date: 2026-05-02.

| Assumption | Project behavior relying on it | Official source | Checked date | Notes/uncertainty |
| --- | --- | --- | --- | --- |
| Custom images can be deployed across regions. | `capture-deploy --execute` captures one custom image in the first requested region, then deploys that image sequentially to each requested region. | [Linode Images guide](https://www.linode.com/docs/products/tools/images/) | 2026-05-02 | Public docs say custom or recovery images can be deployed to any compute region the account can access. Region access and provider-side transfer time remain provider concerns. |
| Image availability is exposed as API status. | Capture waits for the custom image to report `available` before treating it as deployable. | [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json), `GET /images/{imageId}` image `status` | 2026-05-02 | The image status enum includes `creating`, `pending_upload`, and `available`. Region replica status has additional values, but explicit replica management is out of scope. |
| Linode instance running and offline states are exposed as API status. | Deploy waits for `running`; capture waits for `offline` after shutdown before image capture. | [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json), `GET /linode/instances/{linodeId}` Linode `status` | 2026-05-02 | `running` is only provider/API-level readiness. The project does not infer SSH, application, or cloud-init readiness from it. |
| List responses are paginated with page metadata. | Cleanup discovery walks `/linode/instances` by `page` and `pages`, using `page_size=100`. | [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json), pagination envelope and list endpoints | 2026-05-02 | Pagination is required for account-wide cleanup discovery. The project does not assume all resources fit on one page. |
| `401` and `403` both mean the token cannot be used for the requested API operation. | Execute preflight treats either status as `LINODE_TOKEN` rejection and stops before mutating resources. | Linode API error reference, `401` and `403` | 2026-05-02 | Public docs distinguish invalid or expired authentication from authenticated-but-not-authorized access. The project intentionally reports both as token unusable for the workflow. |
| Rate-limit metadata can appear in response headers, and `429` indicates rate limiting. | The API client retries safe read/list/poll operations on `429`; mutation creates and cleanup deletes remain single-attempt. | Linode API rate-limit and response-header references | 2026-05-02 | The current client records sanitized retry events for retry-enabled reads. |
| Linode DELETE success is documented, but retry idempotency after a lost response is not documented. | Cleanup re-fetches eligible candidates before DELETE, sends one DELETE attempt, and reports failed attempts as ambiguous instead of retrying blindly. | [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json), `DELETE /linode/instances/{linodeId}` | 2026-05-02 | The endpoint documents a successful deletion response and destructive effects, but does not explicitly guarantee idempotent retry behavior for lost responses. |
| Tags are first-class fields on the resources the project creates and reads. | The project passes tags on Linode and image creation, validates returned tags, and uses Linode tags for cleanup selection. | [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json), Linode and Image `tags` fields | 2026-05-02 | The API schema exposes tags on Linode and Image resources. Cleanup remains conservative: missing or mismatched tags preserve resources. |
| TTL is project-internal tag data, not provider lifecycle behavior. | Cleanup parses the project-managed `ttl=...` tag and decides whether tagged temporary Linodes are expired. | Project tag contract in `src/linode_image_lab/manifest.py` and cleanup rules in `docs/design.md` | 2026-05-02 | The provider is not expected to expire or delete resources from this TTL. Custom images are preserved unless an explicit future feature changes that contract. |
| Explicit image replication is available but out of scope. | Multi-region capture-deploy relies on cross-region image deployability and does not call `/images/{imageId}/regions`. | [Linode image replication guide](https://www.linode.com/docs/products/tools/images/guides/replicate-an-image/) and [OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json), `POST /images/{imageId}/regions` | 2026-05-02 | Replication can speed direct regional deployment, but adding replica creation, status tracking, or replica cleanup would be a separate feature. |
