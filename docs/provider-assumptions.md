# Provider Assumptions

This log records provider behavior that Linode Image Lab depends on, plus the
project policies chosen when public provider docs are ambiguous. It is a
source-backed checklist for future behavior, docs, and test changes; it is not a
record of live API verification.

Rows carry their own checked dates. Update a row date only when that specific
assumption is rechecked.

| Assumption | Project behavior relying on it | Source | Checked date | Notes/uncertainty |
| --- | --- | --- | --- | --- |
| Custom images can be deployed across regions. | `capture-deploy --execute` captures one custom image in the first requested region, then deploys that image to each requested region with bounded parallel fan-out. | [TechDocs: Images][td-images] | 2026-05-05 | Public docs say a custom or recovery image can be deployed to any compute region the account can access. Region access remains a provider/account concern. |
| Cross-region deploy latency is not specified as a provider contract. | Multi-region capture-deploy waits for each deploy instance to reach provider/API-level `running` status and does not assume a fixed cross-region transfer or boot duration. | [TechDocs: Images][td-images], [TechDocs: Create a Linode][td-create-linode] | 2026-05-02 | Public docs confirm cross-region deployability but do not provide a latency SLA for deploying a captured image into another region. Operators should expect farther-region deploys may take longer. |
| Create Linode accepts an existing firewall id, and firewalls can be read independently. | Deploy preflights a configured firewall with `GET /networking/firewalls/{firewallId}` before creation, then includes `firewall_id` in the create-instance payload only when configured. | [TechDocs: Create a Linode][td-create-linode], [TechDocs: Get a firewall][td-get-firewall] | 2026-05-06 | Firewall creation, mutation, interface generation, and post-create firewall assignment are out of scope. The project treats a missing or inaccessible firewall as a preflight failure before Linode creation. |
| Firewall rules are read and replaced through the firewall rules document, and rule labels/descriptions are operator-facing identifiers. | `firewall-sync` reads `GET /networking/firewalls/{firewallId}/rules`, computes a managed inbound allow rule, and uses `PUT /networking/firewalls/{firewallId}/rules` only with `--execute`. Ownership is exact label plus exact description because no stable per-rule ID is exposed by the rule endpoint. | [TechDocs: List firewall rules][td-get-firewall-rules], [TechDocs: Update firewall rules][td-put-firewall-rules], [TechDocs: Manage firewall rules][td-manage-firewall-rules] | 2026-05-19 | Public docs show rule labels and descriptions are identifiers for account holders and do not affect firewall performance. The project preserves non-managed rules and fails closed when the managed label is ambiguous. |
| Cloud Firewall rule address limits apply to the managed allowlist rule. | `firewall-sync` fails before update when the validated registry would exceed one managed rule's total IPv4 and IPv6 address capacity. | [TechDocs: Cloud Firewalls][td-cloud-firewall], [TechDocs: Manage firewall rules][td-manage-firewall-rules] | 2026-05-19 | Public docs state a per-rule address limit. Splitting a registry across multiple managed rules is out of scope for this first consumer. |
| Object Storage can be accessed through S3-compatible clients with an endpoint URL, access key, and secret key. | `firewall-sync` fetches the registry JSON from the configured endpoint, bucket, and object key using env-only Object Storage credentials. | [TechDocs: Access keys for Object Storage][td-object-access-keys], [TechDocs: Use s4cmd with Object Storage][td-object-s4cmd] | 2026-05-19 | Public docs show region-specific S3 endpoint hostnames and object download by bucket/key plus endpoint URL. The project signs a read-only GET and does not list buckets or objects. |
| Create Linode accepts public SSH authorized keys. | Deploy includes `authorized_keys` in the create-instance payload only when explicitly configured through deploy config or CLI input. | [TechDocs: Create a Linode][td-create-linode] | 2026-05-06 | Public docs describe `authorized_keys` as write-only public SSH keys appended for the root user when deploying from an image. Metadata, user data, authorized users, SSH connectivity validation, and automatic key discovery remain out of scope. |
| Create Linode accepts Base64-encoded metadata user data. | Deploy loads explicit user-data files, Base64 encodes them locally, and includes `metadata.user_data` in the create-instance payload only when configured for deploy instances. | [TechDocs: Create a Linode][td-create-linode], [TechDocs: Add user data when deploying Linodes][td-add-user-data] | 2026-05-06 | Public docs say API and CLI user data must be Base64 encoded. The project does not support inline user-data values, automatic file discovery, semantic cloud-init validation, or applying user data to capture-source Linodes. |
| Image availability is exposed as API status. | Capture waits for the custom image to report `available` before treating it as deployable. Capture-replicate-deploy also requires the captured image to report `available` before submitting replication. | [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json), `GET /images/{imageId}` image `status`; [TechDocs: Get an image][td-get-image] | 2026-05-20 | The image status enum includes `creating`, `pending_upload`, and `available`. Region replica status has additional values and is handled separately for the explicit replication operator path. |
| Linode instance running and offline states are exposed as API status. | Deploy waits for `running`; capture waits for `offline` after shutdown before image capture. | [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json), `GET /linode/instances/{linodeId}` Linode `status`; [TechDocs: Get a Linode][td-get-linode] | 2026-05-02 | `running` is only provider/API-level readiness. The project does not infer SSH, application, or cloud-init readiness from it. |
| Lifecycle waits are bounded by project policy, not provider timing guarantees. | Instance-ready, instance-offline, and image-available waits poll retryable reads until success or the project timeout. | [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json); project implementation in `src/linode_image_lab/linode_api.py` | 2026-05-02 | The current timeout is a project safety bound. A timeout means the tool stopped waiting, not necessarily that the provider resource failed permanently. |
| Custom image creation targets an explicit disk id; disk list ordering is not a selection contract. | Capture lists disks for the temporary source Linode, requires exactly one suitable disk with an integer id, ignores identifiable swap or non-ready disks, and fails before image creation if selection is ambiguous. | [TechDocs: List disks][td-list-disks], [TechDocs: Create an image][td-create-image], [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json) | 2026-05-02 | The image-create request requires `disk_id`. Public List disks docs expose disk information but do not guarantee response ordering, so the project does not infer the boot disk from the first returned item. Disk selection is still limited to the one-suitable-disk case; explicit user-selected disks remain out of scope. |
| List responses are paginated with page metadata. | Cleanup discovery walks `/linode/instances` and `/images` by `page` and `pages`, using `page_size=100`. | [TechDocs: List images][td-list-images], [TechDocs: Pagination][td-pagination], [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json) | 2026-05-14 | Pagination is required for cleanup discovery. The project does not assume all resources fit on one page. |
| `401` and `403` both mean the token cannot be used for the requested API operation. | Execute preflight treats either status as `LINODE_TOKEN` rejection and stops before mutating resources. | [TechDocs: 401][td-401], [TechDocs: 403][td-403] | 2026-05-02 | Public docs distinguish invalid or expired authentication from authenticated-but-not-authorized access. The project intentionally reports both as token unusable for the workflow. |
| Rate-limit metadata can appear in response headers, and `429` indicates rate limiting. | The API client retries safe read/list/poll operations on `429`; mutation creates, shutdown, image create, and cleanup deletes remain single-attempt. Multi-region deploy fan-out is capped so concurrent create attempts remain bounded. | [TechDocs: Rate limits][td-rate-limits], [TechDocs: Response headers][td-response-headers] | 2026-05-05 | Current behavior honors `Retry-After` first, accepting documented numeric seconds and HTTP-date values if present, then `X-RateLimit-Reset`. If neither header is parseable, the client falls back to deterministic backoff and records sanitized retry metadata. Linode documents per-operation rate limits, including a dedicated Create Linode limit, so higher deploy concurrency may still surface provider throttling. |
| Retryable `408` and `5xx` statuses are project policy, not a Linode-specific retry contract. | The API client retries configured read/list/poll operations on `408`, `500`, `502`, `503`, and `504`. | [TechDocs: Errors][td-errors]; project implementation in `src/linode_image_lab/linode_api.py` | 2026-05-02 | Public docs describe standard HTTP error responses, but do not define these statuses as safely retryable for every operation. The project keeps retries bounded and avoids automatic retries for create/shutdown/image POST mutations or DELETE cleanup. |
| Create-instance, create-image, and shutdown POST requests are single-attempt. | `capture --execute`, `deploy --execute`, and `capture-deploy --execute` do not automatically retry Linode create, image create, or shutdown mutations. | [TechDocs: Create a Linode][td-create-linode], [TechDocs: Create an image][td-create-image], [TechDocs: Shut down a Linode][td-shutdown-linode]; project implementation in `src/linode_image_lab/linode_api.py` | 2026-05-02 | This is a mutation-safety policy. Retrying these operations after an ambiguous failure could create duplicate resources or repeat a state-changing action. |
| Linode and image DELETE success is documented, but retry idempotency after a lost response is not documented. | Cleanup re-fetches eligible candidates before standalone `cleanup --execute` DELETE, sends one DELETE attempt, and reports failed attempts as ambiguous instead of retrying blindly. | [TechDocs: Delete a Linode][td-delete-linode], [TechDocs: Delete an image][td-delete-image], [TechDocs: Errors][td-errors]; project implementation in `src/linode_image_lab/cleanup.py` and `src/linode_image_lab/linode_api.py` | 2026-05-14 | The image delete operation warns deletion is irreversible and deletes replicated instances too. Public docs do not specify idempotency or whether an ambiguous network failure means the delete did or did not occur. Deferred run cleanup also keeps DELETE single-attempt after tag-scoped selection. |
| Tags are first-class fields on the resources the project creates and reads. | The project passes tags on Linode and image creation, validates returned tags, and uses Linode tags for cleanup selection. | [TechDocs: Create a Linode][td-create-linode], [TechDocs: Create an image][td-create-image], [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json) | 2026-05-02 | The API schema exposes tags on Linode and Image resources. Cleanup remains conservative: missing or mismatched tags preserve resources. |
| TTL is project-internal tag data, not provider lifecycle behavior. | Cleanup parses the project-managed `ttl=...` tag and decides whether tagged temporary Linodes and discovered lab-owned custom images are expired. | Project tag contract in `src/linode_image_lab/manifest.py` and cleanup rules in `docs/design.md` | 2026-05-14 | The provider is not expected to expire or delete resources from this TTL. Custom images outside the default lab-owned project tag are ignored by standalone cleanup discovery. |
| Explicit image replication is available through a complete image region-set request. | `replicate --execute` submits `POST /images/{imageId}/regions` only after reading the image's existing regions and combining them with requested regions, so a replication request does not intentionally remove existing image regions. | [TechDocs: Replicate an image][td-replicate-image], [Guide: Replicate an image][guide-replicate-image], [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json), `POST /images/{imageId}/regions` | 2026-05-20 | Public docs say requested regions are where the image should be replicated and note that leaving out the original region removes the image from that region. Execute mode fails before mutation if existing regions are not exposed. |
| Explicit image replication target regions must support Object Storage. | `capture-replicate-deploy --execute` reads each requested replication target region and fails before capture if the provider region capabilities do not include `Object Storage`. Deploy regions are checked only when they are also requested replication targets, including the backwards-compatible no-replication-input default. Standalone `replicate --execute` applies the same check before the replication POST. | [Guide: Replicate an image][guide-replicate-image], [TechDocs: Get a region][td-get-region], [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json), Region `capabilities` | 2026-05-20 | The image replication guide states that image storage uses Object Storage and only core compute regions supporting Object Storage are available. A basic `GET /regions/{regionId}` existence check is not sufficient for image replication eligibility; the project requires the capability value exposed by public region data. |
| Public region Object Storage capability is necessary but may not be sufficient for image replication POST acceptance. | `region-policy generate` preserves raw provider capability facts and provider-backed capability groups, but can apply a narrow documented `provider_overrides.image_replication_excluded_regions` table only to generated `country_*_image_replication` helper groups. Execute still validates requested targets and fails closed on provider POST rejection. | [Guide: Replicate an image][guide-replicate-image], [TechDocs: Replicate an image][td-replicate-image], operator smoke results for `us-iad-2`, `de-fra-2`, and `jp-tyo-3` | 2026-05-21 | Public docs state image replication uses Object Storage and that eligible regions support Object Storage, but they do not publish an image-replication-specific exclusion list. Operator smoke found `us-iad-2`, `de-fra-2`, and `jp-tyo-3` advertise Object Storage while image replication should currently exclude them from generated helper groups. The override is temporary provider-discrepancy handling for generated helper groups only, not a mutation of provider facts or an execution fallback. When a country has no remaining valid image-replication regions after documented exclusions, the generated helper group is omitted rather than emitted empty. |
| Replica status values are exposed for custom image regions. | `capture-replicate-deploy --execute` performs a bounded read-only wait after replication submission and deploys only when each requested replication target region reports replica status `available`. Standalone `replicate --execute` still records the submission response and does not poll convergence. | [Guide: Replicate an image][guide-replicate-image], [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json), image `regions[].status` | 2026-05-20 | The guide describes image status moving from pending replication to available, and the OpenAPI schema exposes per-region statuses including `available`, `pending replication`, and `replicating`. The project treats the wait as a bounded readiness gate only; it does not encode timing guarantees, repair replicas, or own regional placement after the run. |
| Public region metadata exposes region ids, capabilities, and country codes without account-specific resource data. | `region-policy generate` stores provider-backed `[provider_regions.*]` facts containing only region ids and capabilities, then generates helper `[generated_groups.*]` from capabilities and provider-exposed country codes. `region-policy validate` compares those facts, supported provider overrides, and helper groups to current provider metadata before trusting operator-owned groups. | [TechDocs: List regions][td-list-regions], [TechDocs: Create Object Storage key workflow][td-create-object-storage-key] | 2026-05-21 | The List regions operation returns region objects, and Object Storage docs instruct operators to verify desired regions include `"Object Storage"` in `capabilities` and store region `id` values. The generated country groups use only provider-exposed two-letter country codes, not city labels, inferred geography, latency, nearest region, fallback placement, or deployment policy. |
