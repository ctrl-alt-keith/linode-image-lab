# Provider Assumptions

This log records provider behavior that Linode Image Lab depends on, plus the
project policies chosen when public provider docs are ambiguous. It is a
source-backed checklist for future behavior, docs, and test changes; it is not a
record of live API verification.

Checked date: 2026-05-05.

| Assumption | Project behavior relying on it | Source | Checked date | Notes/uncertainty |
| --- | --- | --- | --- | --- |
| Custom images can be deployed across regions. | `capture-deploy --execute` captures one custom image in the first requested region, then deploys that image to each requested region with bounded parallel fan-out. | [TechDocs: Images][td-images] | 2026-05-05 | Public docs say a custom or recovery image can be deployed to any compute region the account can access. Region access remains a provider/account concern. |
| Cross-region deploy latency is not specified as a provider contract. | Multi-region capture-deploy waits for each deploy instance to reach provider/API-level `running` status and does not assume a fixed cross-region transfer or boot duration. | [TechDocs: Images][td-images], [TechDocs: Create a Linode][td-create-linode] | 2026-05-02 | Public docs confirm cross-region deployability but do not provide a latency SLA for deploying a captured image into another region. Operators should expect farther-region deploys may take longer. |
| Image availability is exposed as API status. | Capture waits for the custom image to report `available` before treating it as deployable. | [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json), `GET /images/{imageId}` image `status`; [TechDocs: Get an image][td-get-image] | 2026-05-02 | The image status enum includes `creating`, `pending_upload`, and `available`. Region replica status has additional values, but explicit replica management is out of scope. |
| Linode instance running and offline states are exposed as API status. | Deploy waits for `running`; capture waits for `offline` after shutdown before image capture. | [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json), `GET /linode/instances/{linodeId}` Linode `status`; [TechDocs: Get a Linode][td-get-linode] | 2026-05-02 | `running` is only provider/API-level readiness. The project does not infer SSH, application, or cloud-init readiness from it. |
| Lifecycle waits are bounded by project policy, not provider timing guarantees. | Instance-ready, instance-offline, and image-available waits poll retryable reads until success or the project timeout. | [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json); project implementation in `src/linode_image_lab/linode_api.py` | 2026-05-02 | The current timeout is a project safety bound. A timeout means the tool stopped waiting, not necessarily that the provider resource failed permanently. |
| Custom image creation targets an explicit disk id; disk list ordering is not a selection contract. | Capture lists disks for the temporary source Linode, requires exactly one suitable disk with an integer id, ignores identifiable swap or non-ready disks, and fails before image creation if selection is ambiguous. | [TechDocs: List disks][td-list-disks], [TechDocs: Create an image][td-create-image], [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json) | 2026-05-02 | The image-create request requires `disk_id`. Public List disks docs expose disk information but do not guarantee response ordering, so the project does not infer the boot disk from the first returned item. Disk selection is still limited to the one-suitable-disk case; explicit user-selected disks remain out of scope. |
| List responses are paginated with page metadata. | Cleanup discovery walks `/linode/instances` by `page` and `pages`, using `page_size=100`. | [TechDocs: Pagination][td-pagination], [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json) | 2026-05-02 | Pagination is required for account-wide cleanup discovery. The project does not assume all resources fit on one page. |
| `401` and `403` both mean the token cannot be used for the requested API operation. | Execute preflight treats either status as `LINODE_TOKEN` rejection and stops before mutating resources. | [TechDocs: 401][td-401], [TechDocs: 403][td-403] | 2026-05-02 | Public docs distinguish invalid or expired authentication from authenticated-but-not-authorized access. The project intentionally reports both as token unusable for the workflow. |
| Rate-limit metadata can appear in response headers, and `429` indicates rate limiting. | The API client retries safe read/list/poll operations on `429`; mutation creates, shutdown, image create, and cleanup deletes remain single-attempt. Multi-region deploy fan-out is capped so concurrent create attempts remain bounded. | [TechDocs: Rate limits][td-rate-limits], [TechDocs: Response headers][td-response-headers] | 2026-05-05 | Current behavior honors `Retry-After` first, accepting documented numeric seconds and HTTP-date values if present, then `X-RateLimit-Reset`. If neither header is parseable, the client falls back to deterministic backoff and records sanitized retry metadata. Linode documents per-operation rate limits, including a dedicated Create Linode limit, so higher deploy concurrency may still surface provider throttling. |
| Retryable `408` and `5xx` statuses are project policy, not a Linode-specific retry contract. | The API client retries configured read/list/poll operations on `408`, `500`, `502`, `503`, and `504`. | [TechDocs: Errors][td-errors]; project implementation in `src/linode_image_lab/linode_api.py` | 2026-05-02 | Public docs describe standard HTTP error responses, but do not define these statuses as safely retryable for every operation. The project keeps retries bounded and avoids automatic retries for create/shutdown/image POST mutations or DELETE cleanup. |
| Create-instance, create-image, and shutdown POST requests are single-attempt. | `capture --execute`, `deploy --execute`, and `capture-deploy --execute` do not automatically retry Linode create, image create, or shutdown mutations. | [TechDocs: Create a Linode][td-create-linode], [TechDocs: Create an image][td-create-image], [TechDocs: Shut down a Linode][td-shutdown-linode]; project implementation in `src/linode_image_lab/linode_api.py` | 2026-05-02 | This is a mutation-safety policy. Retrying these operations after an ambiguous failure could create duplicate resources or repeat a state-changing action. |
| Linode DELETE success is documented, but retry idempotency after a lost response is not documented. | Cleanup re-fetches eligible candidates before standalone `cleanup --execute` DELETE, sends one DELETE attempt, and reports failed attempts as ambiguous instead of retrying blindly. | [TechDocs: Delete a Linode][td-delete-linode], [TechDocs: Errors][td-errors]; project implementation in `src/linode_image_lab/cleanup.py` and `src/linode_image_lab/linode_api.py` | 2026-05-02 | The DELETE operation documents successful deletion, but public docs do not specify idempotency or whether an ambiguous network failure means the delete did or did not occur. Deferred run cleanup also keeps DELETE single-attempt after tag-scoped selection. |
| Tags are first-class fields on the resources the project creates and reads. | The project passes tags on Linode and image creation, validates returned tags, and uses Linode tags for cleanup selection. | [TechDocs: Create a Linode][td-create-linode], [TechDocs: Create an image][td-create-image], [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json) | 2026-05-02 | The API schema exposes tags on Linode and Image resources. Cleanup remains conservative: missing or mismatched tags preserve resources. |
| TTL is project-internal tag data, not provider lifecycle behavior. | Cleanup parses the project-managed `ttl=...` tag and decides whether tagged temporary Linodes are expired. | Project tag contract in `src/linode_image_lab/manifest.py` and cleanup rules in `docs/design.md` | 2026-05-02 | The provider is not expected to expire or delete resources from this TTL. Custom images are preserved unless an explicit future feature changes that contract. |
| Explicit image replication is available but out of scope. | Multi-region capture-deploy relies on cross-region image deployability and does not call `/images/{imageId}/regions`. | [TechDocs: Replicate an image][td-replicate-image], [Linode OpenAPI schema](https://github.com/linode/linode-api-docs/blob/development/openapi.json), `POST /images/{imageId}/regions` | 2026-05-02 | Replication can speed direct regional deployment, but adding replica creation, status tracking, or replica cleanup would be a separate feature. |
